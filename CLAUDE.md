## What this is

A C++ Viterbi parser for **probabilistic context-free grammars (PCFG)**, applied to Korean verb morphology. It finds the top-_k_ most likely parse trees for a sequence of single-character tokens (`LetterType = char`), given a grammar of weighted productions. The project skeleton was generated by [cmake-init], so most of the `cmake/`, preset, and docs scaffolding is boilerplate ΓÇö the real code lives in `source/` and `test/`.

## Build, run, test

Dependencies (fmt, nlohmann_json, Catch2) are vendored via CMake `FetchContent` ΓÇö no external package manager is required. A network connection is needed on first configure to clone deps.

Plain build:
```sh
cmake -S . -B build -D CMAKE_BUILD_TYPE=Release
cmake --build build
```

Developer mode (enables tests, lint, format, docs, coverage targets) requires `parser_DEVELOPER_MODE=ON`. The presets in `CMakePresets.json` (e.g. `ci-ubuntu`) configure dev mode plus the strict warning/lint flags. The simplest dev test loop:
```sh
cmake -S . -B build -D parser_DEVELOPER_MODE=ON
cmake --build build
cd build && ctest --output-on-failure        # runs Catch2 tests
```

Run a single test by Catch2 tag:
```sh
./build/test/parser_test "[test_basic]"      # or [test_complex]
```

**Important:** `[test_complex]` reads `examples/prods.json` via a *relative path*, so `parser_test` (and `ctest`) must be invoked with the **project root as the working directory**, not from inside `build/`. Run `ctest` from the build dir but `parser_test` directly from the repo root, or `ctest --test-dir build` from root.

Run the executable ΓÇö it reads one JSON request on stdin and writes one JSON result on stdout:
```sh
echo '{"start_symbol":"S","rules":[...],"sentence":"AB","num_trees":1}' | ./build/parser
```

## Architecture

The pipeline is: **JSON grammar ΓåÆ `Pcfg` ΓåÆ `ViterbiParser::parse` ΓåÆ `Tree`s ΓåÆ JSON**.

- **`Nonterminal`** (`nonterminal.hpp`) ΓÇö a named grammar symbol, ordered by name.
- **`Production<TerminalT>`** (`production.hpp`) ΓÇö `lhs ΓåÆ rhs` with a `prob`. `rhs` is a `vector<variant<Nonterminal, Terminal>>`. The whole codebase specializes this as `LetterProd = Production<char>` (alias `LetterType = char` in `pcfg.hpp`).
- **`Pcfg`** (`pcfg.{hpp,cpp}`) ΓÇö holds the start symbol and productions, and precomputes indexes (`Indexes`: lhs/rhs/empty/lexical) and **left-corner relations** (`LeftcornerRelations`, including a transitive closure). Note: these indexes are built but the current `ViterbiParser` does a brute-force scan over all productions and does not yet consult them ΓÇö they exist for a faster parsing strategy.
- **`ViterbiParser`** (`viterbiparser.{h,cpp}`) ΓÇö the core. Bottom-up CYK-style dynamic programming over a sparse `ConstituentMap` keyed by `(begin, end, symbol)`. For each span (lengths 1..n), `add_constituents_spanning` repeatedly applies productions until fixpoint (to handle unary chains), keeping the top-_k_ trees per cell by `log_prob`. `match_rhs` recursively enumerates the ways a production's rhs can tile a span. Probabilities are combined in log space.
- **`Tree`** (`tree.{h,cpp}`) ΓÇö parse tree node (`symbol`, `children` as `variant<Tree, char>`, `log_prob`). Provides `str()` (pretty-print) and `json()` (serialization), plus a `std::hash` specialization so trees can live in `unordered_set`.
- **`main.cpp`** ΓÇö stdin/stdout JSON driver; wraps everything in try/catch and emits `{"status": "success"|"error", ...}`.

### Adding to / changing the grammar interface
The JSON rule format (used by both `main.cpp` and the `[test_complex]` test) is: each rule has `lhs` (string), `prob` (float), and `rhs` (array where an object `{"name": ...}` is a Nonterminal and a bare string's first char is a terminal). `examples/prods.json` is a real Korean-verb grammar fixture.

## Conventions / gotchas

- **Mixed C++ standards:** `parser_lib` is built as **C++20** (it uses `std::span`); `parser_exe` and `parser_test` are C++17. Keep library headers/impl C++20-clean and avoid leaking C++20-only constructs into exe/test translation units.
- The CI build (`CMakePresets.json` / `.github/workflows/ci.yml`) compiles with a very strict warning set (`-Wconversion -Wsign-conversion -Wold-style-cast -Wshadow` ΓÇª as errors via `-Werror` paths) plus clang-tidy and cppcheck. New code should be warning-clean under these.
- Formatting is enforced by `clang-format` (`.clang-format`) and spelling by `codespell` (`.codespellrc`); dev-mode targets `format-fix` / `spell-fix` apply them.
- Header convention is inconsistent in the existing tree (`.hpp` for some files, `.h` for `tree.h`/`viterbiparser.h`) ΓÇö match the file you're editing.
